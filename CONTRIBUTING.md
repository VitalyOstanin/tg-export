## Окружение

Используется `uv` для управления зависимостями и виртуальным окружением.

```bash
# Установка uv: https://github.com/astral-sh/uv
uv sync --extra dev
```

Все команды разработки запускаются через `uv run`, без активации venv напрямую:

```bash
uv run tg-export --help
uv run python -m pytest
```

## Запуск тестов

```bash
uv run python -m pytest
```

Для подробного вывода:

```bash
uv run python -m pytest -v
```

Тесты не обращаются к Telegram API и не требуют сети.

## Формат коммитов

Используется [Conventional Commits](https://www.conventionalcommits.org/ru/v1.0.0/):

```
<тип>(<область>): краткое описание

Опциональное расширенное описание.
```

Типы: `feat`, `fix`, `chore`, `refactor`, `docs`, `test`, `perf`, `ci`.

Примеры из истории:

- `feat(tg): показывать msg_id в выводе messages, дедупликация файлов в download`
- `fix(terminal): эскейп rich-markup в именах чатов/файлов и устранение гонки в Live`
- `chore: подготовка к релизу v1.1.0`

## Pull request

- Один PR -- одна логическая задача. Если нужно несколько правок, разбейте на отдельные PR.
- Все тесты должны проходить перед merge.
- В описании PR кратко: что изменено и зачем, какие тесты добавлены.
- Не добавляйте в репозиторий реальные данные Telegram (имена контактов, чатов, ID, экспортированные файлы).

## Безопасность данных

В коде, тестах, фикстурах, коммитах и любых файлах проекта запрещено использовать:

- реальные имена пользователей, контактов, чатов, групп, каналов;
- реальные Telegram ID;
- реальные api_id/api_hash, session-файлы;
- результаты экспорта `export_output/` (этот каталог и любые другие, содержащие реальные данные).

Используйте только вымышленные данные в примерах и тестах.
